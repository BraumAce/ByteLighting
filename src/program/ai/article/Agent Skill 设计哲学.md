# Agent Skill 设计哲学：从联网场景到通用 Skill 设计方法论

> 参考文章：[一泽Eze - Web Access：一个 Skill，拉满 Agent 联网和浏览器能力](https://mp.weixin.qq.com/s/rps5YVB6TchT9npAaIWKCw)

本文以 Web Access Skill 为切入点，提炼出一套通用的 Agent Skill 设计方法论。前半部分解析联网场景的具体设计，后半部分将这些设计思想泛化到代码审查、调试、数据分析等通用场景，并补充一些个人在实践中的观察和思考。

<!-- more -->

## 一、从一个联网场景说起

### 背景：Agent 联网，远没有看上去那么简单

用过 Claude Code、OpenClaw 等 AI Agent 的人都知道，这些工具自带联网能力。但实际体验下来，你会发现它们的联网水平，大概相当于一个"只会用搜索引擎，不会操作浏览器"的实习生：

- 让它调研小红书上某个产品的风评，它会拿着 Web Search 工具在搜索引擎里反复换关键词，根本拿不到站内内容
- 让它读一个需要 JS 渲染的页面，它用 fetch 拉到一堆空白 HTML，然后告诉你"网站好像挂了"
- 想让它帮你并行查 5 个网站？对不起，串行处理，一个一个来，还可能弹窗抢走你的浏览器焦点

**问题的根源不在于模型不够聪明，而在于缺乏有效的策略指导和完备的工具集。**

<!-- more -->

## 二、核心理念：Skill = 策略哲学 + 最小完备工具集 + 必要的事实说明

在 Agent Skill 的设计中，有一个核心公式：

$$
\text{Skill} = \text{Agent 策略哲学} + \text{最小完备工具集} + \text{必要的事实说明}
$$

这个公式的三个组成部分缺一不可，下面逐一拆解。

### 1. 策略哲学：教 Agent 怎么思考，而不是怎么操作

通用场景的 Skill 设计，需要避免过度指定 Agent "该怎么做"，而是重新校准模型在对应场景的思考方式。

以联网场景为例，设计了一个四步思考循环：

**Step 1 - 定义成功标准**：拿到任务后，先明确什么算完成？需要获取什么信息、执行什么操作、达到什么结果？这是后续所有判断的锚点。

**Step 2 - 选择最优起点**：根据任务性质选一个最可能直达的方式。比如目标涉及小红书、微信公众号等反爬平台，直接进浏览器操作，不在静态工具上浪费时间。

**Step 3 - 过程校验**：每一步的结果都是证据，不只是"成功/失败"的二元信号。搜索没命中，可能不是关键词不对，而是目标本身不存在。重试无改善，说明该换方向了。

**Step 4 - 完成判断**：对照成功标准确认任务完成后停止，不过度操作。

:::tip 关键洞察
这里没有任何具体操作路径，只是把"怎么思考联网任务"描述清楚了。Agent 理解了这个框架，遇到没见过的网站和任务也能做出合理的策略判断。
:::

### 2. 最小完备工具集：搜、看、做

人类上网本质上就三种行为：

| 行为 | 对应工具 | 适用场景 |
|------|---------|---------|
| **搜** | WebSearch | 搜索摘要、发现信息来源 |
| **看** | WebFetch / curl / 浏览器 | 公开页面读取 / 需登录或动态渲染的页面 |
| **做** | 浏览器自动化 | 点击、填表、上传等交互操作 |

Skill 需要做的，是把这些工具整合到位，并清晰描述每个工具的能力边界。Agent 拿到完备的工具集后，结合策略哲学，就能自行判断在什么情况下用什么工具。

:::warning 工具选择的策略
不同工具不是互斥的，而是互补的。静态工具（Search/Fetch）够用时优先使用，遇到障碍时升级到浏览器操作。关键在于 Agent 能自主判断何时切换，而非一条路走到黑。
:::

### 3. 必要的事实说明：激活模型的"惰性知识"

模型几乎知道所有知识，但并非所有知识都能在任务中被第一时间调用。这些"惰性知识"会导致任务执行效率不及预期。

需要在 Skill 中强调的典型惰性知识包括：

- **DOM 边界问题**：Shadow DOM 和 iframe 存在选择器不可跨越的边界，但 eval 递归遍历可以穿透
- **懒加载机制**：提取图片 URL 前需要先滚动到底部触发懒加载，否则部分图片尚未加载
- **反爬误判**：平台返回"内容不存在"不一定是真的不存在，可能是 URL 缺少必要参数或触发了反爬
- **安全边界**：明确告知 Agent 不主动操作用户已有 tab，所有操作在自己创建的后台 tab 中进行

## 三、对抗模型惯性：一个容易被忽视的陷阱

模型有一个很深的惯性思维：

- 联网查数据？那必然用 Web Search
- 搜到这么多站点都这么写？那肯定是事实
- 网站用 fetch 加载不出来？肯定是网站挂了

这种惯性不仅存在于主 Agent，在子 Agent 的任务分配中更加隐蔽。

比如，你让主 Agent "调研小红书上关于 XX 的风评"，主 Agent 分配子任务时，很可能会写成：

> 在小红书上**搜索** XX 相关信息，总结近期风评

"搜索"这个词看似无害，却把子 Agent 锚定到了 WebSearch 上。而小红书是反爬平台，正确的方式是用浏览器直接进站内搜索。

:::danger 用词即暗示
主 Agent 给子 Agent 写 prompt 时，应该描述目标（"获取"、"调研"、"了解"），避免用暗示具体手段的动词（"搜索"、"抓取"、"爬取"）。
:::

## 四、经验沉淀：让 Agent 越用越聪明

Agent 第一次访问一个陌生网站时，需要从头摸索站点结构、URL 模式、反爬策略，步骤多且耗时。但这些信息试过一次就知道了。

一个设计精巧的 Skill 应该包含自动经验沉淀机制：

1. **自动记录**：每次操作完一个站点，自动沉淀平台特征、有效 URL 模式、已知陷阱，按域名存储
2. **自动复用**：下次访问同一站点时，直接加载先验知识，跳过试错环节
3. **标注时效**：经验标注发现日期，视为"可能有效的提示"而非"保证正确的事实"
4. **自动更新**：按经验操作失败时，回退通用模式并更新经验文件

这构成了一个 **Learning Loop**：Agent 的每一次操作都在积累经验，让下一次变得更快更准。

以小红书为例，有经验和无经验模式下的效率差异可达 **10 倍以上**。

## 五、子 Agent 分治：并行能力的正确打开方式

联网任务经常涉及多个独立目标。如果主 Agent 串行处理，不仅慢，所有抓取的中间内容还会涌入主 Agent 的上下文，导致 token 膨胀、推理质量下降。

正确的做法是利用子 Agent 并行分治：

```
主 Agent（决策层）
  ├── 子 Agent 1 → 小红书调研 → 汇总结果
  ├── 子 Agent 2 → 微博调研 → 汇总结果
  ├── 子 Agent 3 → B站调研 → 汇总结果
  └── ...
  └── 最终汇总报告
```

架构上，所有子 Agent 共享同一个浏览器实例和代理，各自创建独立的后台 tab，通过不同的 targetId 操作，互不干扰。

**分治适用性判断：**

| 适合分治 | 不适合分治 |
|----------|-----------|
| 目标相互独立，结果互不依赖 | 目标有依赖关系，下一个需要上一个的结果 |
| 每个子任务量足够大 | 简单单页查询，分治开销大于收益 |
| 需要长时间浏览器操作 | 几次 Search/Fetch 就能完成的轻量查询 |

## 六、泛化思考：从联网到通用 Skill 设计方法论

以上五节拆解了 Web Access Skill 的具体设计。但如果你只关注联网这个场景，就低估了这套设计思想的价值。接下来，我想把视角拉远，从单一场景泛化到通用的 Skill 设计方法论，并补充一些自己在实践中的观察和反思。

### 6.1 核心公式的普适性

回到那个公式：

$$
\text{Skill} = \text{策略哲学} + \text{最小完备工具集} + \text{必要的事实说明}
$$

这三个组成部分，本质上对应的是模型执行任何复杂任务时面临的三个核心挑战：

| 挑战 | 公式对应 | 本质问题 |
|------|---------|---------|
| **不知道怎么思考** | 策略哲学 | 面对开放任务，缺乏结构化的决策框架 |
| **不知道有什么牌可打** | 最小完备工具集 | 工具不全或不知道工具的能力边界 |
| **知道但想不起来** | 必要的事实说明 | 惰性知识在执行时未被激活 |

这三个挑战在联网场景中表现为"不会选工具、不会切换策略、不知道反爬机制"；而在其他场景中，它们同样存在，只是换了一副面孔。

### 6.2 四步思考循环的泛化

Web Access 中的四步循环 —— **定义成功标准 → 选择最优起点 → 过程校验 → 完成判断** —— 不只是一个联网策略，它本质上是一个通用的 **目标导向执行框架**。

我们可以把它映射到几乎所有需要 Agent 自主决策的场景：

**调试（Debugging）场景：**

| 步骤 | 联网场景 | 调试场景 |
|------|---------|---------|
| 定义成功标准 | 拿到什么信息算完成 | 什么表现算修复成功 |
| 选择最优起点 | 搜索还是直接打开浏览器 | 看日志还是加断点还是写最小复现 |
| 过程校验 | 搜索结果是否在推进目标 | 这次修改是否缩小了问题范围 |
| 完成判断 | 对照标准确认完成 | 所有相关测试通过、边界 case 覆盖 |

调试场景中的"模型惯性"同样严重：遇到 bug，Agent 的第一反应几乎总是"加 console.log 看看"。这就好比联网时总想用 WebSearch 一样 —— 不是说 console.log 不好用，而是它不一定是最优起点。如果错误信息已经足够明确（比如一个明显的类型错误），直接定位代码比打日志快十倍。

**代码审查（Code Review）场景：**

- **成功标准**：找出真正影响正确性、安全性、可维护性的问题，而不是挑格式毛病
- **最优起点**：先理解变更的意图（看 PR 描述和 commit message），再看代码实现，而不是上来就逐行阅读
- **过程校验**：每发现一个问题，评估它的严重程度，避免在低优先级问题上花太多时间
- **完成判断**：关键路径是否都覆盖了？是否存在看了代码但没看数据流的盲区？

**数据分析场景：**

- **成功标准**：回答具体的业务问题，而不是生成一堆图表
- **最优起点**：先看数据长什么样（schema、分布、质量），再决定分析方法
- **过程校验**：中间结果是否合理？异常值是数据问题还是真实现象？
- **完成判断**：分析结论是否能直接指导决策？

:::tip 抽象一层来看
四步循环的本质是 **"带着目标做事，用结果校准方向"**。它对抗的是模型最常见的两个问题：（1）没有明确目标就开始执行；（2）执行中遇到阻碍时原地打转而不切换方向。
:::

### 6.3 最小完备工具集：每个领域的"搜、看、做"

Web Access 将联网行为抽象为 "搜、看、做" 三种原子操作。这种抽象方式可以推广到任何领域 —— 关键是找到该领域不可再分的原子行为集合。

**代码开发领域的原子行为：**

| 原子行为 | 对应工具 | 能力边界 |
|----------|---------|---------|
| **读** | Read / Grep / Glob | 读取文件、搜索代码、匹配模式 |
| **写** | Write / Edit | 创建新文件、修改已有文件 |
| **验** | Bash (test/build/lint) | 运行测试、构建检查、代码检查 |
| **查** | Git log / blame / diff | 追溯历史、理解变更上下文 |

少了任何一类，Agent 就会在某些任务上"缺手"。比如没有 git 工具的 Agent 做 bug 修复时，无法追溯问题的引入时间点，只能盲猜。

**系统运维领域的原子行为：**

| 原子行为 | 对应工具 | 能力边界 |
|----------|---------|---------|
| **观测** | 日志查询、监控面板、Metrics | 看到系统当前状态和历史 |
| **诊断** | tracing、profiling、网络抓包 | 定位问题的根因 |
| **操作** | 重启服务、扩缩容、配置变更 | 对系统施加变更 |
| **验证** | 健康检查、流量回放、告警状态 | 确认操作效果 |

:::warning 完备性比丰富性重要
工具不是越多越好。工具过多会增加模型的选择困难，造成"选择瘫痪"。最小完备的含义是：覆盖所有原子行为，但每种行为只提供必要的工具，不做冗余堆叠。当两个工具的能力大幅重叠时，留一个就够了，在 Skill 中说清楚它的适用范围。
:::

### 6.4 惰性知识：每个领域都有自己的"反爬陷阱"

联网场景中的惰性知识是"DOM 边界、懒加载、反爬误判"。其他领域同样存在模型"知道但想不起来"的关键事实。

**代码开发中的惰性知识：**

- **时区问题**：Date 对象在不同环境下的时区行为差异。模型"知道"时区问题存在，但写代码时经常忘记处理
- **浮点精度**：`0.1 + 0.2 !== 0.3`。模型知道这个事实，但在写价格计算逻辑时不一定会主动用 BigDecimal
- **并发安全**：数据库操作的事务隔离级别对并发读写的影响。模型可以解释 MVCC 原理，但在写业务代码时未必会主动加锁或选择合适的隔离级别
- **编码问题**：文件读写时的字符编码，特别是中文环境下的 GBK/UTF-8 混用

**系统运维中的惰性知识：**

- **DNS 缓存**：改了 DNS 记录但不生效，因为各级缓存的 TTL 还没过期
- **连接池耗尽**：服务看起来活着但响应极慢，不是 CPU 或内存的问题，是连接池满了
- **磁盘 inode 耗尽**：`df` 显示磁盘还有空间，但 `df -i` 显示 inode 100%，无法创建新文件

**识别惰性知识的方法论：** 回顾自己与 Agent 协作的历史，找出那些"模型肯定知道，但在这个场景下就是没想起来"的时刻。这些就是需要在 Skill 中显式提醒的惰性知识。一个实用的判断标准：**如果你在 code review 时至少纠正过模型两次同类问题，就值得写进 Skill**。

### 6.5 模型惯性的通用对抗策略

联网场景中的惯性是"总想用 WebSearch"。这种惯性在其他领域同样普遍：

| 领域 | 典型惯性 | 更好的做法 |
|------|---------|-----------|
| 调试 | 遇到 bug 就加 log | 先读错误信息，80% 的 bug 错误信息已经告诉你答案了 |
| 测试 | 只写 happy path 测试 | 先想边界条件和异常路径，再写正常路径 |
| 重构 | 看到重复代码就想抽象 | 先确认是否真的是同一个变化方向（Rule of Three） |
| API 设计 | 直接暴露数据库 schema | 先从调用方的需求出发设计接口 |
| 性能优化 | 直觉优化"看起来慢"的地方 | 先 profiling，用数据说话 |

对抗惯性的核心不是告诉模型"不要做 X"，而是提供一个**更好的思考入口**。以调试为例，不是说"不要加 console.log"，而是说"先花 30 秒读错误信息和调用栈，判断问题类型，再选择对应的调试手段"。这就是策略哲学的价值 —— 它提供了一个"做判断之前先停一步"的思考框架。

### 6.6 Skill 的刚性与弹性光谱

在实践中，不同类型的 Skill 在"刚性（严格遵循）"和"弹性（灵活适应）"之间应该取不同的位置：

```
刚性 ◀━━━━━━━━━━━━━━━━━━━━━━━━━━━━━▶ 弹性

TDD     安全审计     代码审查     联网调研     创意写作
(严格流程)  (合规检查)   (有判断空间)   (高度灵活)   (自由发挥)
```

**刚性 Skill** 适用于有明确最佳实践的领域。TDD 的 "红 → 绿 → 重构" 循环，就是一个不应该被模型"灵活调整"的流程。如果模型"灵活"地跳过了写测试的步骤，直接写实现，那就违背了 TDD 的核心价值。这类 Skill 应该写成严格的工作流程，明确每个步骤和检查点。

**弹性 Skill** 适用于开放性强、场景多变的领域。联网调研就是典型 —— 你不可能预见所有网站的结构和反爬策略，所以只能提供思考框架，让模型自主判断。这类 Skill 应该写成哲学原则，避免规定具体步骤。

:::danger 刚性与弹性错配的代价
- 该刚性的写成弹性 → 模型会"偷懒"跳过关键步骤（比如跳过测试）
- 该弹性的写成刚性 → 模型会机械执行不适用的步骤，在死路上反复撞墙
:::

判断一个 Skill 应该偏刚性还是弹性，可以问一个问题：**这个领域是否存在一条被广泛验证的"黄金路径"？** 如果存在（如 TDD、安全检查清单），偏刚性；如果不存在（如联网调研、创意任务），偏弹性。

### 6.7 经验沉淀的泛化：从站点经验到领域知识库

Web Access 的经验沉淀按域名存储站点操作经验。这个模式可以推广到其他领域：

**代码库经验沉淀：**

每个项目都有自己的"脾气"—— 命名规范、目录结构、常见的坑。Agent 在一个项目上工作久了，应该能积累类似这样的经验：

```markdown
## 项目特征
- 使用 monorepo 结构，packages/ 下有 12 个子包
- API 层使用 tRPC，不是 REST
- 状态管理用 Zustand，不是 Redux

## 有效模式
- 新增 API 路由：在 packages/api/src/routers/ 下创建文件，然后在 index.ts 中注册
- 数据库变更：先写 migration，再更新 Prisma schema，最后 generate

## 已知陷阱
- packages/ui 中的组件不能直接导入 packages/api 的类型，必须通过 packages/shared
- dev 模式下 HMR 偶尔会丢失类型推断，重启 TypeScript server 即可
```

这和站点经验文件的结构几乎一一对应：平台特征 → 项目特征，有效模式 → 有效模式，已知陷阱 → 已知陷阱。

**调试经验沉淀：**

同一类 bug 的修复模式也可以积累。比如"数据库连接超时"这个问题，第一次遇到可能需要排查 30 分钟，但经验沉淀后，Agent 能直接检查连接池配置、网络延迟、慢查询这三个最常见的原因。

:::tip 经验沉淀的通用模式
无论什么领域，经验文件的结构都可以遵循同一个模板：**领域特征（是什么）+ 有效模式（怎么做有效）+ 已知陷阱（什么会失败）+ 时效标注（何时发现的）**。其中时效标注至关重要 —— 它把经验从"绝对真理"降级为"有保质期的参考"。
:::

### 6.8 子 Agent 分治的通用考量

并行分治不仅适用于联网任务。在日常开发中，很多场景都可以受益于子 Agent 并行执行：

**典型的可分治场景：**

- **多文件重构**：5 个独立模块的接口升级，每个模块交给一个子 Agent
- **多维度审查**：安全审查、性能审查、可维护性审查并行进行
- **多平台适配**：iOS、Android、Web 三端的适配逻辑各自独立开发
- **多数据源对账**：同时从 3 个系统拉取数据进行交叉验证

**分治失败的常见原因：**

1. **依赖估计错误**：以为两个子任务独立，实际上共享了一个全局状态（比如数据库 migration 的顺序）
2. **合并冲突**：多个子 Agent 修改了同一个文件的不同部分，合并时产生冲突
3. **上下文不足**：主 Agent 给子 Agent 的 prompt 缺少必要的项目上下文，导致子 Agent 产出的代码风格不一致

分治的关键不是"能不能并行"，而是"合并成本是否低于并行收益"。如果并行完还需要花大量时间统一风格、解决冲突，那还不如串行做。

### 6.9 Prompt 用词的涟漪效应

原文中提到的"用词暗示"问题，值得进一步展开。这不仅仅是联网场景的问题，而是一个在 Agent 交互中无处不在的深层陷阱。

**同一个任务，不同的描述方式，导致完全不同的执行路径：**

| 描述方式 | Agent 被暗示的行为 |
|---------|------------------|
| "优化这个函数的性能" | 微观优化：减少循环、缓存变量、用位运算替代乘除 |
| "让这个接口响应更快" | 系统层面思考：加缓存、优化查询、考虑异步 |
| "用户反馈这个页面很慢" | 全链路思考：前端渲染、网络请求、后端处理、数据库查询 |

三种描述指向的最终目标可能相同，但 Agent 的思考起点完全不同。**用词越具体，Agent 的视野越窄；用词越抽象，Agent 的判断空间越大。**

这与 Skill 设计的核心理念一脉相承：策略哲学之所以要写得抽象，就是为了给模型留出自主判断的空间，而不是用具体的指令锁死它的思考方向。

在实践中，我发现一个有效的经验法则：**在描述任务时，用「结果性动词」替代「手段性动词」**。

- 手段性动词："搜索"、"爬取"、"解析"、"遍历" → 锁定了具体执行方式
- 结果性动词："获取"、"了解"、"确认"、"诊断" → 只定义了期望结果

## 七、设计 Skill 的实操清单

综合以上分析，设计一个高质量 Skill 可以遵循以下清单：

### 第一步：定义 Skill 的位置

- [ ] 明确 Skill 覆盖的场景边界（做什么 / 不做什么）
- [ ] 判断 Skill 在刚性-弹性光谱上的位置
- [ ] 确认目标：是通用能力增强，还是特定工作流标准化

### 第二步：设计策略哲学

- [ ] 提炼该领域的核心思考框架（类似联网场景的四步循环）
- [ ] 识别该领域最常见的模型惯性，设计对抗策略
- [ ] 用抽象原则而非具体步骤来描述策略（除非是刚性 Skill）

### 第三步：梳理最小完备工具集

- [ ] 列出该领域不可再分的原子行为
- [ ] 为每种原子行为匹配对应的工具
- [ ] 明确每个工具的能力边界和适用范围
- [ ] 去除能力重叠的冗余工具

### 第四步：收集必要的事实说明

- [ ] 回顾与 Agent 协作的历史，找出反复出现的"惰性知识遗忘"
- [ ] 只写模型高概率遗忘且影响任务成败的事实
- [ ] 补充安全边界和不可做的事项

### 第五步：设计经验沉淀机制

- [ ] 确定经验的存储粒度（按站点？按项目？按问题类型？）
- [ ] 设计经验的时效标注机制
- [ ] 设计经验失效时的回退和更新策略

## 八、写在最后

回顾全文，Skill 设计的本质可以总结为一句话：

> **好的 Skill 不是教 Agent 怎么做事，而是教 Agent 怎么思考做事。**

Skill 是对人类专家经验的一种结构化表达。一个资深工程师之所以能快速解决问题，不是因为他记得更多的操作步骤，而是因为他有一套成熟的思考框架、完整的工具认知、和大量的领域经验。Skill 要做的，就是把这三样东西用模型能理解的方式传递给 Agent。

从这个角度看，`Skill = 策略哲学 + 最小完备工具集 + 必要的事实说明` 这个公式的真正含义是：

$$
\text{Skill} = \text{思考框架} + \text{能力边界认知} + \text{领域经验}
$$

这也是为什么说 Skill 设计是 Agent 工程中最有杠杆效应的工作 —— 它不改变模型本身的能力，但能显著改变模型能力的释放效率。就像同一个程序员，给他一套好的方法论和工具链，产出可能提升数倍。

最后，回到一个实践层面的建议：不要试图一次性设计出完美的 Skill。好的 Skill 一定是**迭代出来的** —— 先写一个最小版本，在真实任务中使用，观察模型在哪里犯错、在哪里低效，然后针对性地补充策略哲学、工具说明或事实提醒。这本身就是一个 Learning Loop。
