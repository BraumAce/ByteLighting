# 文章正文写作格式

适用于 `src/` 下所有正文文章（不含一级目录 README.md、不含阅读记录——后者另见 [reading-format.md](./reading-format.md)）。

## 章节序号

正文使用 `1.` `1.1` `1.1.1` 编号，对应 `##` `###` `####` 三级标题。

```md
## 1. 章节标题
### 1.1 子章节标题
### 1.2 子章节标题
## 2. 章节标题
### 2.1 子章节标题
```

- 不使用「一、二、三、四」「壹、贰、叁」等中文数字编号
- 不使用「第一章」「第一节」「Chapter 1」称呼
- 不在标题中插入 emoji 装饰、不用 `→` `/` 等符号断句
- 一级标题（`#`）由 frontmatter 的 `title` 渲染，正文不写

## 开篇导读

frontmatter 后的 1-3 段作为开篇导读，最后一段后插入 `<!-- more -->` 分隔正文。首页、分类页、tag 页都以这段导读作为文章卡片的摘要，决定读者是否点开。

- **首选 1 段精简摘要**：一段话定位文章主题与核心论点即可
- **最多 3 段**：确有铺陈必要才扩展，主体内容应放进正文章节
- **不写**「本文将从 X 个维度展开」「先来介绍 Y，再来介绍 Z」这类自述写作动机或结构的句子
- **不写**与主题关系不大的背景铺陈、行业大词、寒暄式引入
- **不写**章节小结、不提前抛最终结论——结论留给正文与文末
- 可选：摘要末尾用 `::: tip` / `::: info` 等容器承载一句金句，但与正文重复时优先合并到摘要正文里，避免「摘要 + tip」表达同一件事

## 行文风格（去 AI 味）

写作目标：让一个工程师读起来像在读另一个工程师的技术总结，而不是在读模型生成的报告。

### 章节衔接

新章节用一两句话承接上一章、定位本节意图，再展开论述。不要每节都自我宣告（「下面我们来看 X」「接下来介绍 Y」）。参考写法：

> 上一节梳理了「问题为什么会发生」，本节回答「怎么知道它正在发生」。

### 强调（粗体使用规则）

粗体的视觉主张是「让读者抓住某个词或某句话」。在「克制」的前提下，下面这几类位置主动用：

- **术语首次定义**：第一次引入领域术语时加粗（如「**最终一致性**」），后续提及不再加粗
- **核心论点 / 总结性句子**：一节里值得让读者带走的那一句加粗，每节 0-2 处
- **对比表格的关键列**：表格用于对比时，分类列 / 术语列加粗便于扫读
- **要点中的关键字**：列表项里的核心动作词或对象，加粗 1-2 个

不要给纯名词复述、整段描述性文字、列表项里的所有关键词加粗。粗体密度高于「每段 1-2 处」就该自查是否在「装饰」而非「强调」。

### 列表与段落

观点连贯的论述写成段落。列表只用于：

- 清单（字段 → 含义、步骤 → 动作）
- 对比（方案 A vs 方案 B 的能力差异）
- 并列要点（彼此独立、不构成因果链）

把一段连贯论述拆成 5 条短列表是 AI 味重灾区。如果几条「列表项」之间有「因为/所以/进而」的关系，那它们应该是一段散文。

### 破折号

`——` 用于补充说明或语义转折，每段最多 1 处；密集的破折号让节奏机械化。优先用句号、分号、换段表达停顿。

### 表格

表格用于对比、映射、清单。不要把段落塞进表格——单元格里如果开始出现 2 句以上的散文，就该改回正文。

### 比喻

关键概念用一个生活类比打通认知。每篇文章 1-3 处即可，不要为了「生动」而强凑。参考：

> 如果 cache 是大脑的短时记忆，database 就是长期记忆。
> 索引是地图，文档是路——没有地图也能找到路，但慢且容易走错。

### 金句

论点可以压缩成一句对仗或递进的话作为节末收束。同样一篇 1-3 处即可，强凑会比没有更糟。

### 收尾

文末段落收束全文要点 + 给读者一个视角拔升或下一步动作。不要写成机械的「综上所述」或「本文介绍了……」。

## 代码与图

### 行内代码

行内代码（`` `xxx` ``）的作用是把「代码符号」与「自然语言」在视觉上隔开，让读者一眼分辨哪些是程序里的标识符、哪些是中文叙述。下列场景一律用行内代码：

| 类别                      | 示例                                                                                       |
| ------------------------- | ------------------------------------------------------------------------------------------ |
| 关键字 / 保留字           | `final`、`return`、`throws`、`synchronized`、`async`、`await`                              |
| 数据类型                  | `int`、`double`、`String`、`HashMap<K, V>`、`Promise<T>`                                    |
| 方法 / API / 函数名       | `length()`、`Arrays.sort()`、`useState`、`fetch()`                                          |
| 运算符与符号              | `==`、`!==`、`+=`、`->`、`?.`、`??`                                                         |
| 类 / 包 / 路径 / 文件名   | `java.util.List`、`src/.vuepress/theme.ts`、`.env.example`                                  |
| 字面量与常量              | `true`、`false`、`null`、`-1`、`'A'`                                                        |
| 注解 / 装饰器             | `@Override`、`@Deprecated`、`@Component`                                                    |
| 占位符 / 格式符           | `%d`、`%s`、`{name}`                                                                        |
| 异常类、错误码、状态码    | `NullPointerException`、`HTTP 404`                                                          |
| 命令行片段                | `npm install -D mermaid`、`git rebase main`                                                 |
| 配置项 key / 环境变量     | `markdown.mermaid`、`OPENAI_API_KEY`                                                        |
| 内联代码片段              | `int x = (int)'A';`、`x = a + b`                                                            |
| 范围 / 区间标注           | `[0, n)`、`-128 ~ 127`                                                                      |

不必用行内代码的场景：

- 普通英文专有名词（OpenAI、VuePress、theme-hope）—— 除非作为包名或命令出现
- 中文术语 / 概念 —— 用粗体或 `「」` 引号
- 整段算法描述 —— 超过一两个 token 时改用代码块

风格细节：

- 同一概念在文中第一次提及时，若属上面 13 类则一律加 backtick；后续保持一致
- 不要过度使用——一句话里塞 5 个以上 backtick 等于没强调
- 行内代码与中文之间的空格规则见下方「行内特殊语法与中文之间留空格」一节

### 代码块

- 必须标语言：` ```python ` ` ```java ` ` ```mermaid ` ` ```typescript ` 等
- 优先真实可跑的代码，不要写「伪代码骨架」式占位
- 同一节内多语言对照用 `::: code-tabs` 切换；同一段代码强调关键行用 ` ```python {2,4-6} ` 高亮

### 流程图与可视化

- 流程、架构、关系优先用 mermaid 画图；超过 3 个节点关系的文字描述应优先考虑画图替代
- 数据展示用 ` ```chart ` / ` ```echarts `；思维导图用 ` ```markmap `（按需开启对应 markdown 配置）

## 行内特殊语法与中文之间留空格

行内的 Markdown 语法符号（粗体 `**`、行内代码 `` ` ``、数学公式 `$`、链接 `[]()`、上下标 `~ ^`、高亮 `==` 等）紧贴中文字符（CJK）时，部分解析器可能识别失败、退化成纯字面量；为了渲染鲁棒性，**符号与中文之间统一留一个半角空格**。紧贴英文字母 / 数字 / 标点符号时不必加。

判定方法：

| 左侧字符 | 右侧字符 | 是否加空格 |
| --- | --- | --- |
| 中文 | 行内语法符号（`**` `` ` `` `$` 等） | 加 |
| 行内语法符号 | 中文 | 加 |
| 行内语法符号 | 标点（`：` `。` `，` `「」` `；`）| 不加 |
| 标点 | 行内语法符号 | 不加（标点本身已是分隔符） |
| 破折号 `——` | 行内语法符号 | 不加（破折号已是分隔符） |
| 英文 / 数字 | 行内语法符号 | 不加（VuePress 自动处理边距） |

正例：

```md
核心原则是 **最终一致性**，强调读多写少场景下……    ✅ 中文与 `**` 之间留空格
对于读多写少场景，请优先考虑 **最终一致性**。      ✅ 粗体后紧贴句号，不加空格
执行 `npm install` 安装依赖。                       ✅ 行内代码两侧空格自然
缓存策略——**写穿透**与回写——是两种典型方案。    ✅ `——` 与 `**` 之间不加空格
公式 $E = mc^2$ 描述质能关系。                       ✅ 行内公式两侧留空格
```

反例：

```md
核心原则是**最终一致性**，强调读多写少场景下……    ❌ 中文紧贴 `**`，部分解析器识别失败
执行`npm install`安装依赖。                          ❌ backtick 与中文紧贴
公式$E = mc^2$描述质能关系。                          ❌ `$` 与中文紧贴
```

此规则同样适用于：

- 表格单元格：`| 中文 **加粗** 中文 |`——单元格内的中文与符号也按同规则留空格
- blockquote、列表项内的行内符号
- 标题里的行内代码

## 提示容器（theme-hope hint container）

theme-hope 提供 7 种 hint container，按内容类型选择，不为视觉效果乱套。

| 类型 | 关键字       | 适用                                            |
| ---- | ------------ | ----------------------------------------------- |
| 提示 | `tip`        | 金句、可记的实践经验、推荐做法                   |
| 信息 | `info`       | 背景知识、上下文补充                             |
| 备注 | `note`       | 离题但有价值的旁白                               |
| 重要 | `important`  | 强约束、必须遵守的规则、设计契约                 |
| 警告 | `warning`    | 反模式、盲区、容易出错的做法                     |
| 危险 | `caution`    | 严重后果、明令禁止的操作                         |
| 折叠 | `details`    | 长代码 / 长引用 / 选读内容                       |

基本语法：

```md
::: tip 自定义标题
内容支持 `代码` 与 [链接](#)。标题可省略，使用类型默认标题。
:::
```

写作建议：

- 一个二级章节里的 hint container 控制在 1-3 个，过多会反向稀释视觉重点
- 反直觉、反模式、容易踩的坑优先用 `warning` / `caution`；金句、推荐实践用 `tip`；强约束用 `important`
- 长引用、可选阅读的扩展内容用 `details`，避免拉长主线行文

## 数学公式（LaTeX）

数学公式、概率统计公式、形式化定义一律走 LaTeX，不要写在 ```text``` 代码块或纯文本里。theme-hope 默认通过 `@vuepress/plugin-markdown-math` 支持 KaTeX。

- 行内：`$E = mc^2$`
- 块级：`$$...$$` 单独成段
- 正文中提到公式里的变量名，也用 `$N$` `$n(q_i)$` 形式包裹，与正文区分清楚

```md
贝叶斯定理：

$$
P(A \mid B) = \frac{P(B \mid A) \, P(A)}{P(B)}
$$

$P(A \mid B)$ 表示在事件 $B$ 发生的条件下事件 $A$ 发生的概率。
```

## 反例与正例

```md
## 一、xxx                                ❌ 中文数字
## 第一章 xxx                             ❌ 章节称呼
## 🚀 性能优化                            ❌ emoji 装饰
## 1. 性能优化                            ✅
### 1.1 缓存策略                          ✅
```

```md
**这一点** **非常** **重要** —— **必须** **优先** **处理**   ❌ 强调与破折号过载
首先 X，其次 Y，最后 Z；总之，要 W              ❌ 机械连接词
缓存策略的第一原则是命中率优先于一致性。           ✅ 自然语句 + 单点强调
```

```md
**反模式**: **务必** **避免** —— **绝不** 在事务里 **同步调用外部 HTTP**     ❌ 强调与破折号过载
::: warning 事务内禁止同步外部调用
在事务里调用外部 HTTP 接口会让连接长时间被占用，高并发时连接池迅速耗尽。
:::                                                                            ✅ 用容器承载警示，正文回归自然
```

```md
公式：P(A|B) = P(B|A) * P(A) / P(B)                          ❌ 数学公式裸写
$$
P(A \mid B) = \frac{P(B \mid A) \, P(A)}{P(B)}
$$                                                            ✅ LaTeX 渲染
```

```md
核心原则是**最终一致性**：读多写少时……                     ❌ 中文紧贴粗体符号，解析风险
核心原则是 **最终一致性**：读多写少时……                    ✅ 中文与粗体之间留空格
```
