# 分类与标签规范

适用于全站 Markdown 文章 frontmatter 中的 `category` 和 `tag` 字段。

## 总原则

1. **优先复用已有**：先在下面的清单里找；找不到合适的再考虑新建。
2. **`category`：1 个为主**，对应文章**直接所属目录**（或最贴近的语义分组）。
3. **`tag`：能少则少**，1 个最佳，1-2 个常见，主题跨多个领域时才超过 2 个。
4. **追求精准而非覆盖**，不堆砌相关词，只保留能让读者准确定位的关键词。

## 选取流程（写新文章时）

```
新文章 → 确定 category（通常 1 个）
        ├─ 文章直接所属目录有对应 category？ → 直接复用
        ├─ 上述清单里有语义匹配的？        → 复用，不新建近义词
        └─ 都没有？                        → 新建（保持与现有命名风格一致）

新文章 → 确定 tag（1～2 个为主）
        ├─ 文章主题最关键的 1 个词         → 必填
        ├─ 跨领域才补第 2 个               → 不为补而补
        ├─ 上述清单里有的优先复用          → 避免 `Agent` / `agent` / `智能体` 这种近义分裂
        └─ 都没有？                        → 新建（保持与现有命名风格一致）
```

## 命名规则

- **大小写**：英文专有名词保持官方写法（`Spring`、`MinIO`、`JUC`、`MyBatis-Plus`、`FastAPI`），不要全小写
- **中英取舍**：已被中文圈广泛使用的中文译名优先（`并发编程`、`系统架构设计师`），新兴未稳定的概念可保留英文（`Prompt`、`Agent`）
- **不带空格符号**：避免 `BFS ` 这种尾空格、避免重复词（`图论` 不要再加 `算法-图论`）
- **不要近义词分裂**：已有 `Agent` 就不要再写 `智能体`；已有 `Prompt` 就不要再写 `提示词`

## 反例

- ❌ `tag: [Java, JUC, 并发编程, 多线程, 线程安全]` —— 5 个 tag，相互高度重叠
- ✅ `tag: [JUC]` —— 一个就够了

- ❌ `category: [Skills, Agent, AI]` —— category 应只 1 个
- ✅ `category: [Skills]`，`tag: [Agent]` —— 主分类对应所属目录，子主题维度作为 tag

- ❌ 新文章随手写 `tag: 智能体`，导致与已有 `Agent` 分裂
- ✅ 直接复用 `Agent`

## 维护

- 当某个 tag/category 累积到一定数量并形成固定主题，考虑提升为目录或固定分类
- 长期只有 1-2 篇文章使用的近义 tag，应在归档时合并
- 新增/合并 tag 时同步更新本文档
